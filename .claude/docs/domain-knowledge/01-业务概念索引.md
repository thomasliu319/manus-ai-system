## 1. 核心业务概念

### 1.1 知识条目 (Knowledge Article)

**定义**：经过采集、分析、整理后的结构化技术资讯单元，是知识库的最小数据单元。

**业务属性**：

| 属性 | 说明 | 示例 |
|------|------|------|
| 唯一标识 (id) | 全局唯一的条目标识符 | `github-20260704-001` |
| 标题 (title) | 技术项目/文章的原始名称 | `langgenius/dify` |
| 数据来源 (source) | 采集来源标识 | `github`, `rss:hackernews` |
| 原始链接 (source_url) | 可追溯到原文的 URL | `https://github.com/...` |
| 技术摘要 (summary) | LLM 生成的中文技术摘要 | 2-3 句话说明核心内容和价值 |
| 质量评分 (score) | 1-10 分的质量评分 | `7` |
| 标签 (tags) | 技术领域标签列表 | `["agent", "llm", "python"]` |
| 受众级别 (audience) | 目标读者的技术水平 | `beginner` / `intermediate` / `advanced` |
| 状态 (status) | 条目在知识库中的生命周期状态 | `draft` / `review` / `published` / `archived` |

**代码位置**：`v2-automation/pipeline/pipeline.py:210-228` (step_organize 字段定义)

---

### 1.2 四步流水线 (4-Step Pipeline)

**定义**：知识加工的标准业务流程，包含采集 → 分析 → 整理 → 保存四个步骤。

| 步骤 | 名称 | 职责 | 输入 | 输出 |
|------|------|------|------|------|
| 1 | 采集 (Collect) | 从外部数据源获取原始数据 | 数据源 API | `knowledge/raw/*.json` |
| 2 | 分析 (Analyze) | LLM 分析每条内容，生成摘要/评分/标签 | 原始数据 | 带分析结果的条目 |
| 3 | 整理 (Organize) | 去重、格式标准化、质量过滤 | 分析后条目 | 标准化知识条目 |
| 4 | 保存 (Save) | 将条目写入文件系统 | 标准化条目 | `knowledge/articles/*.json` |

**代码位置**：
- v2: `v2-automation/pipeline/pipeline.py:261-309` (run_pipeline)
- v3: `v3-multi-agent/workflows/graph.py:27-58` (build_graph)

### 1.3 数据源 (Data Source)

**定义**：知识条目的原始信息来源。

| 数据源 | 采集方式 | 内容特点 |
|--------|----------|----------|
| **GitHub** | Search API (REST) | AI/LLM/Agent 相关的开源仓库，按 star 排序 |
| **RSS Feeds** | HTTP GET + 正则解析 | 12 个技术资讯源，分为 research/industry/open-source/chinese 等类别 |

**RSS 源分类**（`rss_sources.yaml` 配置）：

| 类别 | 源名称 | 内容领域 |
|------|--------|----------|
| general | Hacker News (Best) | 综合技术新闻 |
| research | Reddit r/MachineLearning, arXiv cs.AI | 学术研究 |
| open-source | Reddit r/LocalLLaMA, Hugging Face Blog | 开源社区 |
| industry | OpenAI Blog, Anthropic Research, Google AI Blog | 公司动态 |
| framework | LangChain Blog | 框架生态 |
| chinese | 机器之心, 量子位 | 中文技术媒体 |

**代码位置**：`v2-automation/pipeline/rss_sources.yaml`

### 1.4 质量门控 (Quality Gate)

**定义**：确保知识条目质量的自动化校验机制，包含两个层次。

| 层次 | 工具 | 检查内容 |
|------|------|----------|
| 格式校验 | `hooks/validate_json.py` | 6 必填字段、ID 格式、URL 格式、状态枚举、评分范围 |
| 质量评分 | `hooks/check_quality.py` | 5 维度评分（摘要/深度/格式/标签/空洞词），A/B/C 三级 |

**质量阈值**：
- 格式校验：全部检查项通过 → `exit(0)`
- 质量评分：A (≥80) / B (≥60) / C (<60)，存在 C 级 → `exit(1)`

**代码位置**：`v2-automation/hooks/`

---

### 1.5 审核循环 (Review Loop) — v3 专属

**定义**：LangGraph 工作流中的质量控制机制，分析结果需通过审核才能进入保存阶段。

**审核维度（5 维度加权）**：

| 维度 | 权重 | 说明 |
|------|------|------|
| summary_quality | 0.25 | 摘要是否准确凝练 |
| technical_depth | 0.25 | 分析是否触及技术本质 |
| relevance | 0.20 | 内容与 AI/LLM/Agent 领域的相关度 |
| originality | 0.15 | 内容是否独特 |
| formatting | 0.15 | 标签、分类、摘要格式是否规范 |

**循环规则**：
- 加权总分 >= 7.0 → 通过，进入保存阶段
- 加权总分 < 7.0 → 不通过，带反馈返回 review 重做
- 最多 3 轮，超过后强制通过 + 警告

**代码位置**：`v3-multi-agent/workflows/reviewer.py:15-23`

## 2. Agent 概念

### 2.1 Agent 三角色

| Agent | 角色 | 权限 | 职责 |
|-------|------|------|------|
| **collector** | 数据采集员 | Read, Grep, Glob, WebFetch | 从外部源采集原始数据 |
| **analyzer** | 深度分析员 | Read, Grep, Glob, WebFetch | LLM 分析 + 评分 + 标签 |
| **organizer** | 整理归档员 | Read, Grep, Glob, Write, Edit | 去重、格式化、质量过滤、写入文件 |

**Agent 协作规则**（AGENTS.md 定义）：
1. **单向数据流**：Collector → Analyzer → Organizer，不可反向
2. **职责隔离**：每个 Agent 只操作自己权限范围内的文件
3. **幂等性**：重复运行同一天的采集不应产生重复条目
4. **质量门控**：Analyzer 评分低于 0.6 的条目，Organizer 应丢弃
5. **可追溯**：每个条目保留 `source_url` 和 `collected_at` 用于溯源

---

### 2.2 OpenCode 调用方式

```
@collector 采集今天的 GitHub Trending 数据
@analyzer 分析 knowledge/raw/github-trending-2026-03-17.json
@organizer 整理今天所有已分析的原始数据
```

## 3. 技术概念

### 3.1 MCP (Model Context Protocol)

**定义**：模型上下文协议，允许 AI 工具通过标准化协议读写外部数据源。

**本项目中的应用**：`mcp_knowledge_server.py` 将本地知识库暴露为 MCP 服务，使 OpenCode 等 AI 工具可以通过 `search_articles` / `get_article` / `knowledge_stats` 工具查询知识库。

**协议版本**：JSON-RPC 2.0 over stdio，协议版本 `2024-11-05`

---

### 3.2 LangGraph（v3）

**定义**：基于状态图的工作流编排框架，用于构建可审核、可重试的多步骤 AI 流水线。

**本项目中的应用**：替代 v2 的过程式 `run_pipeline()`，以声明式图结构定义工作流拓扑，增加审核循环节点。

---

### 3.3 LLM 提供商

项目中支持的三种 LLM 提供商，通过 OpenAI 兼容 API 统一调用：

| 提供商 | API Base | 默认模型 | 用途 |
|--------|----------|----------|------|
| DeepSeek | `api.deepseek.com` | `deepseek-chat` | 默认分析引擎 |
| Qwen (通义千问) | `dashscope.aliyuncs.com` | `qwen-plus` | 备选分析引擎 |
| OpenAI | `api.openai.com` | `gpt-4o-mini` | 备选分析引擎 |

**代码位置**：`v2-automation/pipeline/model_client.py:153-178`

## 4. 数据生命周期

### 4.1 条目状态流转

```
         ┌──────────┐
         │  draft   │  ← 初始草稿（采集/分析失败时的兜底状态）
         └────┬─────┘
              │ (LLM 分析成功)
              ▼
         ┌──────────┐
         │  review  │  ← 待审核（默认状态，等待人工/自动审核）
         └────┬─────┘
              │ (审核通过 / v3 review_passed=true)
              ▼
         ┌──────────┐
         │published │  ← 已发布（最终可检索状态）
         └────┬─────┘
              │ (过期/不再维护)
              ▼
         ┌──────────┐
         │archived  │  ← 已归档（保留但不主动展示）
         └──────────┘
```

### 4.2 数据文件生命周期

```
外部 API → knowledge/raw/raw_{ts}.json  (原始采集)
                │
                ▼
         LLM 分析 → 内存中的 analyses 列表
                │
                ▼
         去重 + 标准化 → 内存中的 articles 列表
                │
                ▼
         knowledge/articles/{id}.json  (持久化条目)
```

## 5. 业务术语速查表

| 术语 | 英文 | 含义 |
|------|------|------|
| 采集 | Collect / Collection | 从外部源抓取原始数据 |
| 分析 | Analyze / Analysis | LLM 对内容进行理解、评分、分类 |
| 整理 | Organize | 去重、格式化、质量过滤 |
| 保存 | Save / Persist | 将条目写入文件系统 |
| 审核 | Review | 质量评估与循环改进（v3） |
| 知识条目 | Knowledge Article | 标准化的技术内容单元 |
| 原始数据 | Raw Data | 未处理的 API 响应数据 |
| 相关性评分 | Relevance Score | 0-1 或 1-10 的质量评分 |
| 标签 | Tag | 技术领域分类标签 |
| 受众 | Audience | 目标读者技术水平（beginner/intermediate/advanced） |
| 质量门控 | Quality Gate | 自动化格式校验与内容评分 |
| 去重 | Deduplication | 按 source_url 去除重复条目 |
| 指数退避 | Exponential Backoff | 重试等待时间指数增长（1s→2s→4s） |
| 兜底值 | Fallback Value | LLM 调用失败时使用的默认值 |
| 审核反馈 | Review Feedback | 审核未通过时的改进意见 |
| 成本追踪 | Cost Tracking | 按节点累计 LLM Token 用量和费用 |
