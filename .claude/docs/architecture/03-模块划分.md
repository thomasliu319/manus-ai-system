# 03 - 模块划分

> 本文档描述项目的模块边界、模块内聚度和模块间耦合关系。

---

## 1. 模块总览

```
manus-ai-system/
├── v1-skeleton/          # 骨架版：OpenCode Agent 手动驱动
├── v2-automation/        # 自动化版：Python CLI 脚本驱动（当前主版本）
└── v3-multi-agent/       # 多智能体版：LangGraph 图驱动 + 设计模式库
```

各版本之间的关系：
- **v2 继承 v1** 的 Agent 定义和 skill 定义
- **v3 继承 v2** 的 pipeline 模块、hooks 模块和 MCP 模块
- **v3 新增** workflows/ 和 patterns/ 模块

---

## 2. v2 模块划分

### 2.1 模块一览

| 模块 | 路径 | 内聚度 | 对外耦合 |
|------|------|--------|----------|
| pipeline | `pipeline/` | 高 | httpx, yaml, dotenv |
| hooks | `hooks/` | 高 | 无（仅标准库） |
| MCP Server | `mcp_knowledge_server.py` | 高 | 无（仅标准库 + knowledge/） |
| OpenCode 配置 | `.opencode/` | 中 | OpenCode IDE 运行时 |
| 知识数据 | `knowledge/` | — | 被 pipeline/hooks/MCP 读写 |

### 2.2 pipeline/ 模块内部结构

```
pipeline/
├── pipeline.py           # 主编排：四步流水线 + CLI
├── model_client.py       # LLM 客户端：工厂模式
├── rss_reader.py         # RSS 采集
└── rss_sources.yaml      # RSS 源配置
```

**模块内依赖关系**：
```
pipeline.py ──import──→ model_client.py
pipeline.py ──import──→ rss_reader.py
rss_reader.py ──read──→ rss_sources.yaml
model_client.py ──read──→ .env
```

### 2.3 hooks/ 模块内部结构

```
hooks/
├── validate_json.py      # JSON 格式校验（独立，CLI 调用）
└── check_quality.py      # 5 维度质量评分（独立，CLI 调用）
```

两个 hook 文件互相独立，均不依赖 pipeline 模块。它们读取 `knowledge/articles/` 下的 JSON 文件，执行校验/评分后输出结果。

### 2.4 MCP Server 模块

```
mcp_knowledge_server.py   # 独立进程，stdio 通信
```

独立于 pipeline 运行，仅读取 `knowledge/articles/` 目录。与 pipeline 在数据层共享文件系统，无代码级耦合。

## 3. v3 新增模块

### 3.1 workflows/ 模块

```
workflows/
├── graph.py               # 状态图组装 + 编译
├── state.py               # 共享状态定义（KBState TypedDict）
├── nodes.py               # 5 个节点函数实现
├── reviewer.py            # 审核节点（5 维度加权评分）
└── model_client.py        # v3 轻量版 LLM 客户端
```

**模块内依赖关系**：
```
graph.py ──import──→ nodes.py (collect/analyze/organize/save)
graph.py ──import──→ reviewer.py (review)
graph.py ──import──→ state.py (KBState)

nodes.py ──import──→ state.py (KBState)
nodes.py ──import──→ workflows/model_client.py (chat, chat_json)

reviewer.py ──import──→ state.py (KBState)
reviewer.py ──import──→ workflows/model_client.py (chat_json, accumulate_usage)
```

**节点职责**：

| 节点 | 文件 | 函数 | 输入 | 输出 |
|------|------|------|------|------|
| collect | nodes.py | `collect_node()` | — | sources |
| analyze | nodes.py | `analyze_node()` | sources | analyses |
| review | reviewer.py | `review_node()` | analyses[:5] | review_passed + feedback |
| organize | nodes.py | `organize_node()` | analyses | articles |
| save | nodes.py | `save_node()` | articles | 文件写入 |

### 3.2 patterns/ 模块

```
patterns/
├── router.py              # Router 模式：两层意图分类 + 三种分发
└── supervisor.py          # Supervisor 模式：Worker-Supervisor 审核循环
```

**patterns 与 workflows 的区别**：
- `patterns/`：可复用的设计模式实现，独立于具体工作流
- `workflows/`：具体的生产工作流实现，使用 LangGraph 编排

**patterns 模块依赖**：
```
router.py ──import──→ pipeline/model_client.py (chat)
supervisor.py ──import──→ pipeline/model_client.py (chat)
```

注意：patterns 依赖 v2 的 `pipeline/model_client.py`，而非 v3 的 `workflows/model_client.py`。

## 5. 模块设计原则

| 原则 | 实践 |
|------|------|
| **单一职责** | 每个模块只做一件事（采集 / 分析 / 校验 / 检索） |
| **依赖倒置** | LLM 调用通过 `LLMProvider` 抽象接口，不直接耦合具体提供商 |
| **数据层共享** | 各模块通过 `knowledge/` 目录以文件系统作为数据总线通信 |
| **向后兼容** | v2 的 `pipeline.py` 可通过重导出使用 `rss_reader.py` 的函数 |
| **错误隔离** | 单个采集源失败不影响其他源；单个条目解析失败使用默认值兜底 |
| **可替换性** | LLM 提供商可通过环境变量/CLI 参数替换，无需修改代码 |
