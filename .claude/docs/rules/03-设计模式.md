## 1. 工厂模式（Factory Pattern）

### 应用场景

封装多 LLM 提供商的创建逻辑，通过环境变量切换而不修改调用代码。

### 实现

**代码位置**：`v2-automation/pipeline/model_client.py:165-191`

```python

    """
    工厂函数：创建 LLM 客户端（默认使用 DeepSeek）。
    支持 deepseek / qwen / openai 三种提供商。
    """
```

**提供商配置映射**（`model_client.py:153-178`）：

| 提供商 | API Key 环境变量 | 默认 Base URL | 默认模型 |
|--------|------------------|---------------|----------|
| deepseek | `DEEPSEEK_API_KEY` | `https://api.deepseek.com` | `deepseek-chat` |
| qwen | `QWEN_API_KEY` | `https://dashscope.aliyuncs.com/compatible-mode/v1` | `qwen-plus` |
| openai | `OPENAI_API_KEY` | `https://api.openai.com/v1` | `gpt-4o-mini` |

### 类结构

```
LLMProvider (ABC)
  └── OpenAICompatibleProvider   # 统一实现 OpenAI Chat Completions 格式
```

所有三家提供商共用 `OpenAICompatibleProvider` 实例，因为 DeepSeek、Qwen、OpenAI 都遵循相同的 API 格式。通过工厂函数注入不同的 `api_key`、`base_url`、`model` 来区分。

## 2. 策略模式（Strategy Pattern）

### 应用场景

流水线中不同数据源的采集策略（GitHub API vs RSS Feed）。

### 实现

**代码位置**：`v2-automation/pipeline/pipeline.py`

```python
def step_collect(sources: list[str], limit: int) -> list[dict[str, Any]]:
    all_items = []
    if "github" in sources:
        all_items.extend(collect_github(limit))   # GitHub 策略
    if "rss" in sources:
        all_items.extend(collect_rss(limit))      # RSS 策略
```

每个采集策略返回统一的 `list[dict[str, Any]]` 格式，上游 `step_collect` 无需知道具体实现细节。

### 策略实现

| 策略 | 函数 | 数据获取方式 |
|------|------|-------------|
| GitHub | `collect_github()` | GitHub Search API (REST) |
| RSS | `collect_rss()` | HTTP GET + 正则解析 XML |

## 4. 重试模式（Retry Pattern）— 指数退避

### 应用场景

LLM API 调用失败时的容错处理。

### 实现

**代码位置**：`v2-automation/pipeline/model_client.py:199-231`

```python
def chat_with_retry(provider, messages, ..., max_retries=3, backoff_base=2.0):
    for attempt in range(max_retries):
        try:
            return provider.chat(...)
        except (HTTPStatusError, ConnectError, TimeoutException) as e:
            wait_time = backoff_base ** attempt  # 指数退避：1s, 2s, 4s
            time.sleep(wait_time)
```

### 参数说明

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `max_retries` | 3 | 最大重试次数 |
| `backoff_base` | 2.0 | 退避基数（秒），第 n 次重试等待 `backoff_base^n` 秒 |

## 5. 路由模式（Router Pattern）

### 应用场景

用户意图识别与分发：根据用户输入路由到不同的处理器。

### 实现

**代码位置**：`v3-multi-agent/patterns/router.py`

**两层路由策略**：

```
用户输入
  │
  ├── 第一层：关键词快速匹配（零 LLM 成本）
  │   └── 匹配成功 → 直接分发到对应处理器
  │
  └── 第二层：LLM 分类（处理模糊意图）
      └── LLM 返回意图类型 → 分发到对应处理器
```

### 意图 → 处理器映射

| 意图 | 处理方式 |
|------|----------|
| `github_search` | GitHub Search API (`urllib.request`) |
| `knowledge_query` | 本地 `knowledge/articles/` 检索 |
| `general_chat` | LLM 直接回答 |

## 6. 监督者模式（Supervisor Pattern）

### 应用场景

Worker 输出 → Supervisor 审核 → 不通过则重做，确保输出质量。

### 实现

**代码位置**：`v3-multi-agent/patterns/supervisor.py`

```
Task → Worker 生成 → Supervisor 评分
                       │
                       ├── score >= 7 → 返回结果
                       └── score < 7  → 带反馈重做（最多 3 轮）
                                          └── 超过 3 轮 → 强制返回 + 警告
```

### 审核维度

| 维度 | 评分范围 |
|------|----------|
| 准确性 (accuracy) | 1-10 |
| 深度 (depth) | 1-10 |
| 格式 (format) | 1-10 |

## 7. 管道模式（Pipeline Pattern）

### 应用场景

四步流水线：采集 → 分析 → 整理 → 保存。

### 实现（v2）

**代码位置**：`v2-automation/pipeline/pipeline.py:261-309`

```python
def run_pipeline(sources, limit, dry_run, steps):
    if 1 in run_steps: raw_items = step_collect(sources, limit)
    if 2 in run_steps: analyzed_items = step_analyze(raw_items)
    if 3 in run_steps: organized_items = step_organize(analyzed_items)
    if 4 in run_steps: saved_files = step_save(organized_items, dry_run=dry_run)
```

### 实现（v3 — LangGraph 版）

**代码位置**：`v3-multi-agent/workflows/graph.py:27-58`

```
START → collect → analyze → review → organize ─(passed)→ save → END
                                       ↑         │
                                       └─────────┘ (not passed)
```

v3 版本在 v2 基础上增加了 `review` 节点和条件循环边。

## 8. 状态模式（State Pattern — LangGraph）

### 应用场景

通过共享状态对象在各工作流节点间传递数据，避免全局变量。

### 实现

**代码位置**：`v3-multi-agent/workflows/state.py`

```python
class KBState(TypedDict):
    sources: list[dict]        # 采集层
    analyses: list[dict]       # 分析层
    articles: list[dict]       # 整理层
    review_feedback: str       # 审核层
    review_passed: bool        # 审核结果
    iteration: int             # 审核循环计数
    cost_tracker: dict         # 成本追踪
```

每个节点函数签名为 `KBState → dict`（部分状态更新），LangGraph 自动合并。

### 报告式通信原则

> 每个字段都是下游节点可直接消费的结构化摘要，不包含原始日志、API 响应、中间计算碎片或调试信息。

**代码位置**：`v3-multi-agent/workflows/state.py:1-7`

## 10. 设计模式总结

| 模式 | 所在文件 | 核心价值 |
|------|----------|----------|
| 工厂模式 | `pipeline/model_client.py` | 多提供商切换，对调用方透明 |
| 策略模式 | `pipeline/pipeline.py` | 多数据源统一采集接口 |
| 模板方法 | `pipeline/pipeline.py` | LLM 响应解析流程标准化 |
| 重试模式 | `pipeline/model_client.py` | API 调用容错 |
| 路由模式 | `patterns/router.py` | 意图识别零成本方案 |
| 监督者模式 | `patterns/supervisor.py` | 输出质量保障 |
| 管道模式 | `pipeline/pipeline.py` / `workflows/graph.py` | 步骤化数据处理 |
| 状态模式 | `workflows/state.py` | 节点间结构化通信 |
| 便捷函数 | `pipeline/model_client.py` | 降低 API 使用门槛 |
